Description: Detects if calls to `current_setting()` and `auth.<function>()` in RLS policies are being unnecessarily re-evaluated for each row

The lint item titled "Auth RLS Initialization Plan" identifies a performance issue with the row-level security (RLS) policy named "Users can delete their own prescriptions" on the public.prescriptions table. The current implementation calls auth.<function>() for each row, which can lead to suboptimal query performance. The recommended fix is to replace auth.<function>() with (select auth.<function>()), which allows the Postgres optimizer to cache the results per-statement rather than evaluating the function for each row.

Suggested Fixes

  1. Modify the RLS Policy: Update the existing RLS policy to use the select statement for calling the auth.<function>(). This change will improve performance by reducing the number of function calls made during row evaluations.

Here is the SQL to alter the policy:

SQL Query

ALTER POLICY "Users can delete their own prescriptions" 
ON public.prescriptions 
FOR DELETE 
TO authenticated 
USING ((select auth.uid()) = user_id);

Explanation

  • This SQL command alters the existing policy to use (select auth.uid()) instead of auth.uid(), which optimizes the performance of the RLS policy by caching the result of the function call.
  • Ensure that the user_id column is indexed to further enhance performance when this policy is evaluated.
